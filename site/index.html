JavaScript and TypeScript tooling,
implemented in pure Common Lisp.

Clun is a runtime, package manager, test runner, bundler, and HTTP stack for JS/TS. The entire implementation is Common Lisp — not a wrapper around Node, V8, or libuv. JavaScript is only what you run.

On the shipped capability surface, Clun meets or exceeds Bun, Node.js, and npm (evidence-backed Yes / Partial / No in the matrix below).

What ships

Familiar commands and APIs. Underneath: a from-scratch ECMAScript engine, pure-CL TLS, package install from the public npm registry, and first-party tooling — all Common Lisp.

Runtime

Run .js / .ts / JSX. TypeScript strip plus structural typecheck. Node-compat surface for common modules.

Package manager

clun install, add, remove, publish — npm-compatible dependencies over pure-CL HTTPS with SRI verification.

Test runner

clun test with Jest-style APIs, async tests, filtering, and concurrent files.

HTTP and streams

First-party server and router. Streaming request and response bodies; WebSocket + Pub/Sub; no HTTP/2 server. CompressionStream / DecompressionStream (gzip/deflate).

Build and tooling

Bundler, monorepo workspaces, hot reload, formatter/linter, single-file executables.

Platform APIs

SQL drivers, Redis, S3-compatible client, cookies, secrets vault, password hashing, glob, semver — without installing a separate toolchain for each.

Write ordinary JavaScript

Handlers and tests look like Bun or the web platform. The process is Common Lisp.

hello.ts 
const project: string = "Clun";
const runtime = "Common Lisp";

console.log(`${project} runs on ${runtime}`);
const server = Clun.serve({
  port: 3000,
  fetch: async (req) => {
    return new Response("ok");
  },
});
import { expect, test } from "bun:test";

test("adds", () => {
  expect(1 + 1).toBe(2);
});
await Clun.write("out.txt", "hello");
const text = await Clun.file("out.txt").text();

Why Common Lisp

Clun is not a thin shell over another JavaScript engine. Lexer, parser, object model, modules, async, package manager, TLS, and servers are written in ANSI Common Lisp on SBCL, with pure-Lisp libraries vendored in-tree.

  1. 1
    From-scratch engine

    No V8, no SpiderMonkey, no QuickJS foreign link for the runtime itself.

  2. 2
    Purity gate

    make purity fails closed on CFFI / foreign-library slips in Clun’s own sources.

  3. 3
    Vendored and pinned

    A source build does not pull Quicklisp at build time or npm for Clun itself.

  4. 4
    TLS in Lisp

    HTTPS for installs and clients uses the pure-CL TLS stack; CI runs make test-tls.

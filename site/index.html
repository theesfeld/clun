JavaScript tooling.
Implemented entirely in Common Lisp.

Clun is a from-scratch JS/TS runtime, npm-style package manager, test runner, bundler, and HTTP toolkit — written in pure Common Lisp. No Node underneath, no V8, no C, no CFFI. Familiar Bun-shaped commands and APIs; evidence-backed Yes on most of Bun’s public capability matrix, and exceeds Bun on several ledger rows.

Linux + macOS · x64 + arm64

Install Clun

View shell script
$ curl -fsSL https://clun.sh/install | sh

Verifies SHA-256, stages a full versioned bundle, and switches the stable ~/.local/bin/clun launcher. Pin with INSTALL_VERSION; override the bin dir with INSTALL_DIR; control PATH edits with ADD_PATH=0 / ADD_PATH=1.

The hosted command still installs verified v0.2.0-dev.6. Source is the v0.2.0-dev.7 candidate (secrets Yes + colored CLI). Until then, the command only reinstalls v0.2.0-dev.6. Built-in clun --check-update / clun --update. Prefer build from source for the current toolkit.

Pre-alpha: pre-1.0 versions may include breaking changes.

Why this exists

Same jobs as Bun — without a JavaScript implementation underneath.

Run TypeScript, install packages from the public npm registry, run tests, and serve HTTP with commands that feel like the modern JS toolkit. The difference: every line of the implementation is Common Lisp. That is the product, not a footnote.

What you use day to day

Run, install packages, store secrets, test, and serve — in Common Lisp.

npm package management and encrypted secrets storage are first-class product surfaces, not side notes. Commands mirror the Bun/npm-era toolkit so existing JS project habits transfer. Every tool below is pure Common Lisp with repository evidence for the claimed surface.

01$ clun app.ts

JavaScript & TypeScript runtime

A from-scratch engine in Common Lisp runs your JS, ESM/CJS, JSON, and TypeScript.

  • TypeScript strip + transforms; clun tsc structural typecheck (Bun has no typecheck)
  • JSX/TSX, modules, timers, promises, files, fetch, URL
  • Node-shaped builtins for the covered matrix (path, fs, buffer, crypto, …)
  • Proxy traps and invariants are implemented for the covered paths
Built in Pure Common Lisp · no V8 · no Node
02$ clun install [pkg…]

npm package management

Critical path. Install packages the way you already do: add a dependency, resolve the graph, write a lockfile, fill node_modules.

  • clun add / clun install <pkg> create a minimal package.json when the directory has none
  • Public npm registry over pure-CL TLS; SRI checks; offline reinstall from clun.lock
  • Workspaces / monorepo filters when you need them
  • Hosted v0.2.0-dev.6 ships this path — try clun add picocolors
  • No lifecycle scripts (install hooks never run)
Shipped Public add/install on v0.2.0-dev.6
Still open npm publish and full registry-auth corpus remain Partial on the ledger
03Clun.secrets

Encrypted secrets storage

Critical path. Bun-shaped get/set/delete plus has/list/clear backed by a pure-CL AES-256-GCM vault — no CFFI, no OS keychain required for the vault API.

  • Object and positional argument forms match the Bun-shaped surface
  • Persistence, wrong-key denial, and suite receipts in-repo
  • Exceeds Bun’s API with list/clear helpers on the vault
Shipped Pure-CL AES-256-GCM vault (ledger Yes)

05$ clun server.ts

HTTP & WebSocket

Clun.serve with fetch-style handlers, streaming bodies, WebSockets, and pure-CL TLS for clients.

  • HTTP/1.1 serve + router helpers
  • Request / Response / Headers / AbortController
  • Streaming request and response bodies; WebSocket + Pub/Sub; no HTTP/2 server
TLS Pure Common Lisp stack is experimental and unaudited

Familiar APIs · pure Lisp underneath

Write JavaScript the way you already do.

Handlers look like Bun or the web platform. The process executing them is Common Lisp. Unsupported behavior fails loudly instead of silently pretending.

hello.ts 
const project: string = "Clun";
const runtime = "Common Lisp";

console.log(`${project} runs on ${runtime}`);
const server = Clun.serve({
  port: 3000,
  fetch: async (request: Request) => {
    return Response.json({ ok: true });
  },
});

console.log(server.url);
test("math stays honest", () => {
  expect(2 + 2).toBe(4);
});
const file = Clun.file("message.txt");

Clun.write(file, "hello from Clun\n")
  .then(() => file.text())
  .then((text) => console.log(text));

The implementation is the point

Common Lisp all the way down.

Not a thin wrapper around Node. Not a foreign-function bridge to V8. ANSI Common Lisp + SBCL + vendored Lisp libraries only — CFFI shortcuts for Clun itself are rejected (make purity); user native addons load through a pure-CL host with a narrow allowlisted load/call boundary.

  1. 01
    From-scratch engine

    Lexer, parser, object model, modules, async, and runtime APIs are all in this repository.

  2. 02
    Vendored and pinned

    A fresh source build pulls no Quicklisp packages and no runtime npm for Clun itself.

  3. 03
    Purity is checked

    CI runs make purity so CFFI / foreign-library slips cannot land quietly.

  4. 04
    TLS is pure Lisp too

    HTTPS for package installs and clients uses Clun’s own TLS stack. It is experimental and unaudited — good enough for development and evidence gates, not a claim of browser-grade WebPKI.
